# 니어립 (2026.03.31 ~ 2026.04.17)

## 소개

니어립은 원하는 책을 검색한 뒤, 해당 책을 소장한 주변 도서관과 현재 대출 가능 여부까지 확인할 수 있는 서비스입니다.

- 서비스 주소: [https://nearlib.com](https://nearlib.com)

## 서비스가 해결하는 문제

원하는 책을 도서관에서 빌리려면 먼저 내 주변에 어떤 도서관이 있는지 찾아야 하고, 각 도서관 홈페이지에서 다시 같은 책을 검색해 소장 여부를 확인해야 했습니다.

책이 있더라도 지금 바로 빌릴 수 있는지는 또 별도로 확인해야 했고, 도서관마다 검색 방식과 제공 정보가 달라 여러 곳을 비교하기도 어려웠습니다. 결국 원하는 책을 빌릴 수 있는 도서관을 찾기 위해 검색과 확인 과정을 여러 번 반복해야 했습니다.

저 또한 취업 준비 중 기술 서적이나 원하는 책을 도서관에서 자주 빌려 봤는데, 이 반복적인 확인 과정이 번거로웠습니다. 니어립은 이 문제를 해결하기 위해 책 검색부터 지역 기준 도서관 확인, 대출 가능 여부 확인까지를 하나의 서비스 안에서 제공합니다.

## 핵심 기능

- 책 제목이나 저자명으로 원하는 도서를 찾을 수 있습니다.
- 검색 결과를 다시 불러오거나 다른 조건으로 바로 다시 검색할 수 있습니다.
- 책 상세 정보와 대출 관련 정보를 함께 확인할 수 있습니다.
- 선택한 지역 안에서 해당 책을 가진 도서관을 찾을 수 있습니다.
- 도서관 목록과 지도를 함께 보며 위치와 기본 정보를 비교할 수 있습니다.
- 특정 도서관 기준으로 지금 대출 가능한 상태인지 확인할 수 있습니다.

## 주요 사용자 플로우

### 1. 읽고 싶은 책이 이미 정해져 있을 때

사용자는 홈에서 책 제목이나 저자명으로 검색을 시작하고, 도서 검색 결과 페이지에서 원하는 책을 선택합니다.

이후 `소장 도서관 찾기` 버튼을 눌러 지역 선택 다이얼로그를 열고, 선택한 지역을 기준으로 도서관 검색 결과 다이얼로그에서 해당 책을 소장한 도서관을 확인합니다.

### 2. 검색 결과가 여러 권이라 어떤 책인지 다시 확인해야 할 때

사용자는 도서 검색 결과 페이지에서 같은 제목의 다른 책이나 판본 후보를 비교하고, 필요하면 `상세 보기` 다이얼로그를 열어 저자, 출판 정보, 대출 관련 정보를 자세히 확인합니다.

찾는 책이 맞다고 판단한 뒤에만 `소장 도서관 찾기`로 넘어가 지역을 선택하고 도서관을 확인합니다.

### 3. 도서관은 찾았지만 지금 바로 빌릴 수 있는지가 궁금할 때

사용자는 지역을 고른 뒤 열린 도서관 검색 결과 다이얼로그에서 목록과 지도를 함께 보며 위치와 기본 정보를 비교합니다. 이후 특정 도서관을 선택하고 `대출 가능 여부 조회`를 눌러, 해당 도서관에서 지금 바로 빌릴 수 있는지 확인합니다.

## 아키텍처

니어립은 사용자가 보는 화면(web), 외부 도서 Open API 호출 및 응답 가공 서버(bff), 그리고 web과 bff가 함께 참조하는 공통 응답 타입을 분리한 `pnpm` workspace 기반 모노레포입니다.

- `apps/web`
  - 홈 검색, 도서 검색 결과 페이지, 상세 보기 다이얼로그, 지역 선택 다이얼로그, 도서관 검색 결과 다이얼로그 등 사용자가 직접 보는 화면과 상호작용을 담당합니다.
  - 외부 도서 Open API를 직접 호출하지 않고 서버의 `/api` 엔드포인트만 사용합니다.
- `apps/bff`
  - 외부 도서 Open API를 대신 호출하고, 인증키는 서버 환경 변수에서만 관리합니다.
  - 외부 응답을 그대로 반환하지 않고 화면에 필요한 값만 가공해 클라이언트에 전달합니다.
- `packages/contracts`
  - web과 bff가 함께 쓰는 공통 요청 값과 응답 타입을 관리합니다.
  - 서버와 클라이언트가 서로 다른 응답 구조를 기대하지 않게 합니다.

클라이언트 환경 변수는 번들에 포함되어 인증키를 숨길 수 없지만 서버 환경 변수는 서버 런타임에서만 관리되고 외부 요청도 서버에서 직접 보낼 수 있었기 때문에, 프론트엔드 전용 서버(bff)를 사이에 위치시켜 클라이언트가 Open API를 직접 호출하지 않고 bff만 호출하도록 구현했습니다.

외부 응답을 그대로 클라이언트에 전달하면 UI가 화면 표시에 필요한 값을 직접 변환해야 했기 때문에, 서버에서 화면에 필요한 값만 가공해 전달했습니다. 클라이언트는 외부 응답 구조나 데이터 변환 방식을 알 필요 없이 같은 형태의 데이터를 기준으로 화면을 표시합니다.

서버와 클라이언트가 응답 타입을 각각 따로 관리하면 필드명 변경이나 객체 수정이 한쪽에만 반영될 수 있었기 때문에, 같은 저장소 안에서 공통 응답 타입을 함께 참조할 수 있는 모노레포 구조를 선택했습니다.

## 요구 환경

- Node.js 24 이상
- pnpm 10 이상

현재 저장소는 Node `24.12.0`, pnpm `10.28.0` 환경에서 확인했습니다.

## 빠른 시작

1. 의존성을 설치합니다.

```bash
pnpm install
```

2. 예시 파일을 기준으로 환경 변수를 준비합니다.

```bash
cp apps/web/.env.example apps/web/.env
cp apps/bff/.env.example apps/bff/.env
```

3. 아래 값을 채웁니다.

- `apps/web/.env`
  - `VITE_API_BASE_URL`: bff 주소
  - `VITE_KAKAO_MAP_APP_KEY`: 카카오 지도 키
- `apps/bff/.env`
  - `WEB_APP_ORIGIN`: 운영 기준 web origin
  - `LIBRARY_API_BASE_URL`: 외부 도서 Open API 주소
  - `LIBRARY_API_AUTH_KEY`: 외부 도서 Open API 인증키

로컬 개발에서는 `apps/bff/.env`의 `ALLOW_DEV_CORS_ORIGINS=true`로 두면 `http://localhost:5173`, `http://127.0.0.1:5173` 요청을 임시로 허용할 수 있습니다.

4. web과 bff를 함께 실행합니다.

```bash
pnpm dev:all
```

- web: `http://127.0.0.1:5173`
- bff: `http://127.0.0.1:3001`

## 주요 스크립트

- `pnpm dev`
  - web만 실행합니다.
- `pnpm dev:bff`
  - bff만 실행합니다.
- `pnpm dev:all`
  - web과 bff를 함께 실행합니다.
- `pnpm build`
  - web build를 실행합니다.
- `pnpm build:bff`
  - bff build를 실행합니다.
- `pnpm build:all`
  - workspace 전체 build를 실행합니다.
- `pnpm typecheck:all`
  - workspace 전체 타입 점검을 실행합니다.
- `pnpm test:web`
  - web 테스트를 watch 모드로 실행합니다.
- `pnpm test:run:web`
  - web 테스트를 1회 실행합니다.

## 환경 변수

- web 예시: `apps/web/.env.example`
- bff 예시: `apps/bff/.env.example`

주의할 점:

- web의 `VITE_API_BASE_URL`은 외부 API가 아니라 BFF를 가리켜야 합니다.
- 외부 도서 Open API 인증키는 BFF 환경 변수에서만 관리합니다.
- 카카오 지도 키는 web 환경 변수에서 설정합니다.
- `WEB_APP_ORIGIN`은 `https://` 기반의 정확한 origin 값이어야 합니다.
