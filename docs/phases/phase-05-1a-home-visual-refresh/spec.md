# Phase 5-1A. 홈 메인 화면 스크린샷 정합 리디자인

## 목표

- 현재 홈 메인 화면을 `stitch_design_specification_implementation.zip`의 참조안과 최대한 비슷한 인상으로 재구성한다.
- `Phase 5-1`에서 구현한 검색 시작 기능과 제출 계약은 유지하고, 홈 첫 화면의 비주얼 구조와 정보 위계를 다시 설계한다.
- 구현자가 “어디까지 스크린샷을 따라해야 하는지”를 별도 판단하지 않도록, 참조 우선순위와 허용 범위를 문서로 고정한다.

## 1차 source of truth

- 루트 `.impeccable.md`
- `docs/phases/phase-04-2-ux-ui-design/spec.md`
- 루트 `AGENTS.md`
- `stitch_design_specification_implementation.zip`
  - `screen.png`: 최우선 시각 정합 기준
  - `code.html`: 구조와 비율 구현 힌트
  - `DESIGN.md`: 표면 계층, 타이포 톤, 모션 원칙 기준

## 구현 범위

- 홈 route의 히어로 구조와 검색 시작 블록의 시각 레이아웃을 리디자인한다.
- `HomePage`와 `features/book` 검색 시작 UI의 비율, 표면 계층, 타이포, 카피 밀도, support 영역 시각 위계를 정리한다.
- 홈 화면에 한해 스크린샷 정합도를 높이기 위한 전용 레이아웃/스타일 조정을 허용한다.
- 홈 route integration test와 관련 문서를 리디자인 결과에 맞게 갱신한다.

## 비범위

- 도서 검색 결과 화면 구현
- 지역 선택, 도서관 결과, 지도 연동
- 검색 시작 feature의 제출 계약 변경
- 앱 전역 폰트 시스템 전면 교체
- `Phase 5-2` 이후 기능 구현

## 현재 기반 상태

- `pages/home/ui/HomePage.tsx`는 `h1 + BookSearchStart`만 조합하는 얇은 route shell 구조다.
- `features/book`는 세그먼트 탭, 단일 입력, 검색 CTA, support 영역, canonical 제출 계약까지 구현돼 있다.
- 현재 홈 화면은 기능은 갖췄지만, 히어로 비중과 검색 블록 인상이 참조 스크린샷보다 평평하고 카드형 느낌이 강하다.

## 리디자인 계약

### 1. 참조 우선순위

- 시각 비율과 전체 인상은 `screen.png`를 최우선으로 따른다.
- 구체적 구조와 배치 방식은 `code.html`을 참고한다.
- 표면 계층, 경계선 사용 금지, 블루 강조 톤은 `DESIGN.md`를 따른다.
- 한국어 카피, 접근성, 반응형, 기능 경계는 현재 프로젝트 문서를 우선한다.

### 1-1. 스크린샷에서 그대로 따라갈 요소

- 화면 중앙에 대형 히어로 제목과 보조 문단, 검색 슬랩이 수직으로 정렬된 구성
- 히어로 제목이 보조 문단보다 압도적으로 큰 비율을 가지는 점
- 제목 내 강조 구간이 블루와 기울임으로 분명히 구분되는 방식
- 검색 슬랩이 화면 대비 좁고, 크게 떠 있는 단일 인터랙션 블록처럼 보이는 점
- 슬랩 상단의 작은 세그먼트 탭 레일과 하단 한 줄 검색 행 구조
- 입력 행 오른쪽에 짧고 강한 CTA가 붙는 비율
- 두꺼운 구분선 없이 밝기 차이와 부드러운 그림자로 레이어를 표현하는 방식

### 1-2. 프로젝트 문맥상 바꿔야 하는 요소

- 영문 제목, 영문 본문, `Archive Quest`, `Silent Knowledge` 같은 브랜드 카피는 한국어 서비스 문맥에 맞게 교체한다.
- 스크린샷의 폰트 패밀리를 그대로 복제하지 않고, 현재 프로젝트 폰트 안에서 최대한 비슷한 비율과 인상을 만든다.
- 현재 구현된 support 영역과 예시 검색어는 기능상 유지하되, 스크린샷에는 없는 보조 영역으로 낮은 위계에 배치한다.
- 스크린샷에 없는 접근성 semantics, label, helper text, 키보드 이동 계약은 프로젝트 기준을 우선한다.
- 스크린샷처럼 보이더라도 검색 제출 계약과 feature 경계는 `Phase 5-1` 구현 상태를 유지한다.

### 1-3. 정합도 판단 기준

- 구현자는 “느낌만 비슷한가”가 아니라, 제목 비율, 중앙 정렬, 검색 슬랩 폭, 탭 레일 위치, 입력/CTA 한 줄 구조가 스크린샷과 가까운지 기준으로 판단한다.
- 홈 화면의 첫인상은 현재보다 “기능 카드”가 아니라 “에디토리얼 히어로 + 하나의 검색 장치”에 가까워야 한다.
- support 영역은 시선 흐름상 제목과 검색 슬랩 뒤에 따라오는 보조 정보로 읽혀야 한다.

### 2. Home 히어로 구조

- 홈 화면은 중앙 정렬된 히어로 레이아웃으로 재구성한다.
- 요소 순서는 아래로 고정한다.
  - 대형 `h1`
  - 짧은 보조 문단
  - 단일 검색 슬랩
  - support 영역
- `h1`은 스크린샷처럼 2행 이상으로 읽히는 에디토리얼 구조를 가진다.
- 제목 일부는 블루 강조와 기울임 또는 동등한 대비 방식으로 처리한다.
- 보조 문단은 중앙 정렬, 제한된 최대 너비, 2줄 내외 밀도로 유지한다.
- 검색 시작 블록은 히어로 아래에서 별도의 두 번째 카드 군집이 아니라, 하나의 좁고 집중된 슬랩처럼 보이는 폭으로 유지한다.

### 3. 검색 슬랩 구조

- 검색 시작 블록은 큰 카드 패널이 아니라, 하나의 떠 있는 검색 슬랩처럼 보여야 한다.
- 슬랩 내부 구조는 아래로 고정한다.
  - 상단 작은 세그먼트 레일
  - 하단 한 줄 검색 행
- 검색 행은 검색 아이콘, 입력, CTA가 한 시선 흐름 안에서 읽혀야 한다.
- 입력과 CTA는 같은 행에 붙어 있어야 하며, CTA는 스크린샷처럼 오른쪽 끝의 짧고 강한 액션 블록이어야 한다.
- CTA 라벨은 스크린샷처럼 짧은 단일 행동 문구를 사용한다.
- 외곽 구획은 두꺼운 1px 실선보다 밝기 차이, 약한 유령 테두리, 부드러운 그림자로 표현한다.

### 4. support 영역 위계

- `도서 검색 시작` 같은 내부 섹션 제목은 히어로 집중도를 깨면 제거하거나 시각적으로 크게 약화한다.
- 안내 문단과 예시 검색어는 검색 슬랩 아래의 보조 정보 영역으로 이동한다.
- 예시 검색어는 카드형 섹션이 아니라 작은 칩/버튼 수준으로 유지한다.
- support 영역은 검색 행보다 시각 우선순위가 낮아야 한다.
- support 영역은 별도 카드 박스보다 여백과 작은 컨트롤 밀도로 분리하는 것을 우선한다.

### 5. 표면 계층과 타이포

- 홈 배경은 밝고 차분한 표면 위에, 검색 슬랩만 떠 있는 듯한 계층을 만든다.
- 구획 표현의 기본값은 톤 차이와 깊이감이며, 선은 최소화한다.
- 제목은 현재 홈보다 훨씬 더 큰 비율과 강한 대비를 가진다.
- 본문은 제목 대비 확실히 눌러서 조용하게 유지한다.
- CTA는 제한된 블루 그라디언트 또는 동등한 깊이감으로 스크린샷의 “짧고 강한 버튼” 인상을 따른다.

### 6. 반응형과 접근성

- 데스크톱에서는 스크린샷처럼 중앙 집중형 구성을 유지한다.
- 모바일에서는 검색 슬랩이 첫 fold 안에서 보이도록 제목과 간격을 재조정한다.
- 기존 `tablist`, `tab`, 입력 label, helper text, 키보드 이동 계약은 그대로 유지한다.
- 리디자인은 접근성 semantics를 약화시키지 않아야 한다.

## 구현 기준

- `HomePage`는 계속 route shell만 담당한다.
- `BookSearchStart`는 계속 검색 시작 feature의 public API를 유지한다.
- 리디자인 구현에는 `frontend-design`을 기본으로, `arrange`, `distill`, `typeset`, `clarify`, `adapt`, `harden`, `polish`를 함께 사용한다.
- React 구조 변경이 필요하면 `vercel-react-best-practices` 기준으로 역할 분리와 읽기 쉬움을 우선한다.

## 완료 기준

- 홈 화면이 `screen.png`와 최대한 비슷한 중앙 히어로 + 단일 검색 슬랩 인상을 가진다.
- 검색 시작 기능은 기존과 동일하게 동작한다.
- support 영역이 주인공이 아니라 보조 정보로 읽힌다.
- 홈 리디자인 결과가 시스템 다크모드에서도 같은 위계와 집중도를 유지한다.
- 홈 route integration test, lint, typecheck, build가 모두 통과한다.
- 문서만 읽고 구현자가 “무엇을 얼마나 스크린샷처럼 맞춰야 하는지” 설명할 수 있어야 한다.
