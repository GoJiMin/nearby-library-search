# Phase 4-2. UX/UI 설계 기준 정리

## 목표

- Phase 5 기능 구현 전에 MVP 전체 사용자 흐름의 UX/UI 기준을 결정 완료 상태로 고정한다.
- 홈 검색 시작, 도서 선택, 지역 선택, 결과 확인까지의 화면 구조와 정보 위계를 문서화한다.
- 라이트/다크 테마, 색상, 타이포, 간격, 상태 피드백, 반응형 기준을 공통 계약으로 정리한다.
- Phase 5 구현자가 추가 디자인 결정을 하지 않고도 바로 화면 구현에 들어갈 수 있는 design handoff spec을 만든다.

## 기술 결정

- 디자인 컨텍스트 source of truth
  - 루트 `.impeccable.md`
  - 루트 `AGENTS.md`의 `Design Context`
- 설계 톤: 절제된 게임감
- 설계 범위: MVP 전체 화면
- 산출물 성격: 문서 설계 전용, 코드 구현은 포함하지 않음
- Phase 5 handoff 기준
  - screen inventory
  - state matrix
  - layout contract
  - theme/token contract
  - interaction/motion contract
  - copy and CTA contract

## 구현 범위

- 홈 검색 시작 화면의 첫 행동 구조와 핵심 CTA를 정의한다.
- 도서 검색 결과와 도서 선택 화면의 정보 구조를 정의한다.
- 지역 선택 다이얼로그의 단계, 활성/비활성 규칙, 상호작용 흐름을 정의한다.
- 도서관 결과 화면의 지도/목록 조합과 화면별 레이아웃 계약을 정의한다.
- 로딩, 빈 상태, 오류 상태를 전체 플로우 기준으로 정리한다.
- 라이트/다크 테마, 토큰, 타이포, 간격, 밀도, 모션, 접근성 기준을 정리한다.
- HomePage audit에서 드러난 설계 문제를 문서 단계에서 선제적으로 해소한다.
- Phase 5 구현을 위한 `task.md` 작성 기준을 함께 정리한다.

## 비범위

- `apps/web`의 실제 페이지/컴포넌트 구현
- 전역 스타일 토큰과 공통 UI 컴포넌트의 실제 수정
- 카카오맵 SDK 연결, API 호출, 기능 상태 관리 구현
- Phase 5 feature/task를 대신 수행하는 코드 작업
- 디자인 툴 원본 제작이나 외부 프로토타이핑 툴 문서화

## 현재 기반 상태

- 루트 `.impeccable.md`와 `AGENTS.md`에 사용자, 브랜드, 미감, 접근성 기준이 정리돼 있다.
- 현재 `HomePage`는 설명 중심 placeholder 성격이 강하고, 실제 첫 행동 UI가 없다.
- 현재 전역 스타일은 밝은 배경, 블루 액센트, 둥근 패널, 카드 중심 구조를 사용하고 있다.
- `HomePage` audit 기준으로 다음 문제가 이미 확인됐다.
  - 첫 행동 UI 부재
  - 중첩 카드와 반복 카드 패턴
  - 보조 텍스트 대비 부족
  - 다크 모드 계약 부재
- `entities/book`, `entities/library`, `entities/region`는 Phase 5 조합에 필요한 읽기 모델을 이미 제공한다.

## UX/UI 상세 설계

### 1. 경험 원칙

- 첫 행동은 즉시 보여야 한다.
- 설명보다 조작을 먼저 배치한다.
- 각 화면은 하나의 핵심 질문과 하나의 핵심 CTA를 가진다.
- 게임 같은 느낌은 진행감, 선택 피드백, 결과 도달감에만 제한적으로 사용한다.
- 실제 업무성 정보 전달은 항상 명확하고 과장되지 않게 유지한다.

### 2. Screen Inventory

- `Home / Search Start`
  - 책 제목 입력
  - 최근/예시 검색 보조 정보 또는 짧은 안내
  - 검색 시작 CTA
- `Book Search Result / Book Select`
  - 검색어 기준 도서 목록
  - 도서 메타 정보
  - 선택 CTA
  - 검색 결과 없음 상태
- `Region Selection Dialog`
  - 시/도 선택
  - 세부 지역 선택
  - 선택 완료 CTA
  - 지역 결과 없음 상태
- `Library Result`
  - 도서 요약 정보
  - 지도
  - 도서관 목록
  - 정렬/상태 배지
  - 도서관 결과 없음 상태
- `Shared States`
  - 전체 화면 로딩
  - 부분 로딩
  - API 오류
  - 잘못된 검색/선택 조합

### 3. Flow Contract

- 홈 진입 시 사용자는 즉시 검색 입력과 CTA를 볼 수 있어야 한다.
- 검색 실행 후 결과가 1건 이상이면 도서 선택 단계로 이동한다.
- 검색 결과가 1건뿐이어도 선택 확인 과정을 유지할지 여부는 Phase 5에서 다시 정하지 않고, Phase 4-2 문서에서 명시적으로 고정한다.
- 도서가 선택되면 지역 선택 다이얼로그를 연다.
- 지역 선택은 상위 지역 선택 전까지 세부 지역을 비활성 상태로 둔다.
- 지역 선택 완료 후 도서관 결과 화면으로 이동한다.
- 결과 화면은 지도와 목록을 함께 제공하되, 모바일에서는 순차 정보 구조를 우선한다.

### 4. Screen Contract

#### Home / Search Start

- 첫 화면 최우선 요소는 설명 배너가 아니라 검색 입력이다.
- 헤드라인은 사용자의 과업을 바로 설명해야 하며, 부연 문구는 한 문단 이내로 제한한다.
- 반복 카드 3개를 동일 비중으로 나열하는 패턴은 사용하지 않는다.
- “어떻게 동작하는지” 설명은 보조 영역으로 축소한다.
- 빈 상태 카드로 랜딩 화면을 대체하지 않는다.

#### Book Search Result / Book Select

- 사용자는 검색 결과에서 “어떤 책을 선택해야 하는지” 빠르게 판단할 수 있어야 한다.
- 각 아이템은 제목, 저자, 출판 관련 보조 정보, 선택 CTA를 가진다.
- 동일한 카드 반복이 필요하더라도 선택 우선순위가 시각적으로 드러나야 한다.
- 모바일에서는 목록 집중, 데스크톱에서는 선택과 보조 설명을 병행할 수 있다.

#### Region Selection Dialog

- 상위 지역 선택 전에는 세부 지역 입력이 비활성 상태여야 한다.
- 현재 선택 상태는 시각적으로 분명히 드러나야 한다.
- 선택 완료 CTA는 유효한 지역 조합이 만들어진 뒤에만 활성화한다.
- 단계감은 주되, 과도한 게임 UI나 장식적 진행 바는 도입하지 않는다.

#### Library Result

- 화면의 주인공은 “지도”가 아니라 “의사결정에 필요한 정보”다.
- 데스크톱은 지도와 목록을 동시 노출하는 2영역 구성을 기본으로 한다.
- 모바일은 도서 요약, 핵심 상태, 목록, 지도 순 또는 지도 토글 구조 중 하나로 고정한다.
- 지도와 목록은 같은 데이터를 서로 다른 시각화 방식으로 보여줘야 한다.

### 5. State Matrix Contract

- `loading`
  - 다음 행동이 기다림 상태라는 점이 분명해야 한다.
  - skeleton, 단계 안내, 진행감 중 무엇을 쓰는지 문서에서 고정한다.
- `empty`
  - “아무것도 없음”이 아니라 다음 행동을 안내해야 한다.
  - 검색 결과 없음, 지역 결과 없음, 도서관 결과 없음은 서로 다른 메시지를 가져야 한다.
- `error`
  - 문제 상황, 복구 행동, 다음 이동 경로가 함께 보여야 한다.
- `disabled`
  - 지역 선택 전 세부 지역, 유효하지 않은 선택 상태 등 제약 이유가 사용자에게 설명돼야 한다.

### 6. Theme / Token Contract

- 라이트와 다크 모두 설계 범위에 포함한다.
- 다음 토큰 그룹을 Phase 4-2에서 문서로 고정한다.
  - background
  - surface
  - surface-muted
  - text
  - text-muted
  - line
  - accent
  - accent-soft
  - focus
- 일반 본문 텍스트와 보조 텍스트는 WCAG AA 대비를 만족해야 한다.
- 보조 텍스트 토큰은 현재보다 더 높은 대비를 목표로 한다.
- 테마 전환 시 카드, 배경, 라인, 오버레이, 다이얼로그, 지도 보조 패널까지 함께 고려한다.

### 7. Typography / Spacing Contract

- 화면마다 다른 텍스트 크기 체계를 임의로 만들지 않는다.
- display, section, item title, body, caption 수준의 계층을 문서로 고정한다.
- 여백은 컴포넌트 단위가 아니라 화면 리듬 단위로 정의한다.
- 랜딩, 다이얼로그, 결과 화면 모두 동일한 밀도 체계를 공유해야 한다.

### 8. Interaction / Motion Contract

- 기본 경험은 빠르고 차분해야 한다.
- 모션은 상태 전환과 선택 피드백에만 사용한다.
- 과장된 bounce/elastic 계열 움직임은 사용하지 않는다.
- 절제된 게임감은 아래에 한정한다.
  - 단계 전환의 명확한 피드백
  - 선택 완료 시 짧은 보상감
  - 결과 도달 시 시선 유도
- reduced motion을 해치지 않는 수준으로 문서화한다.

### 9. Responsive Contract

- 데스크톱과 모바일은 같은 화면을 단순 축소하지 않는다.
- 모바일에서는 첫 행동과 현재 상태가 항상 fold 상단에서 파악돼야 한다.
- 결과 화면은 좁은 화면에서 지도와 목록이 충돌하지 않는 구조로 고정한다.
- `100vh` 중심 레이아웃은 모바일 브라우저 주소창 변화에 안전한 방식으로 재검토한다.

### 10. Accessibility Contract

- 모든 주요 텍스트는 WCAG AA 대비 이상을 목표로 한다.
- heading 구조는 시각 표현과 문서 구조가 일치해야 한다.
- 입력, 다이얼로그, 선택 상태는 키보드와 스크린리더 기준으로 설계한다.
- 포커스 링, 비활성 상태 설명, 에러 메시지 위치를 문서로 고정한다.

## Handoff 산출물 기준

- Phase 4-2 `spec.md`만으로 다음을 판단할 수 있어야 한다.
  - 각 화면의 핵심 목적
  - 각 화면의 1차 CTA
  - 각 상태에서 보여줄 정보
  - 데스크톱/모바일 레이아웃 차이
  - 라이트/다크에서 바뀌는 토큰
  - 접근성 및 모션 제약
- Phase 5 구현자는 추가 디자인 결정을 하지 않고 `task.md`를 그대로 구현할 수 있어야 한다.

## 산출물

- Phase 4-2 `spec.md`
- Phase 4-2 `task.md`
- `.impeccable.md`와 충돌하지 않는 MVP UX/UI 설계 기준

## 완료 기준

- MVP 전체 화면과 상태가 모두 문서로 정의돼 있다.
- HomePage audit의 P0/P1 문제를 설계 단계에서 해소하는 기준이 반영돼 있다.
- 라이트/다크, 타이포, 간격, 상태, 반응형, 접근성 규칙이 최소 구현 단위까지 명시돼 있다.
- Phase 5 구현자가 추가 디자인 결정을 하지 않아도 되는 수준으로 문서가 결정 완료 상태다.

## 테스트 기준

- 문서만 읽고 각 화면의 핵심 CTA를 설명할 수 있어야 한다.
- 문서만 읽고 검색, 도서 선택, 지역 선택, 결과 확인 플로우를 이어서 구현할 수 있어야 한다.
- 문서만 읽고 로딩/빈 상태/오류 상태를 서로 다르게 설계할 수 있어야 한다.
- `.impeccable.md`와 상충되는 브랜드/미감 기준이 없어야 한다.

## 기본 가정

- Phase 4-2는 코드 구현이 아니라 문서 설계 전용 단계다.
- 디자인 컨텍스트는 `.impeccable.md`와 `AGENTS.md`를 기준으로 유지한다.
- 게임감은 절제된 수준으로만 사용하고, 실용성과 빠른 UX를 우선한다.
- 실제 토큰 수정과 UI 구현은 Phase 5 또는 후속 구현 task에서 수행한다.
