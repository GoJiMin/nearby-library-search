# Phase 4-2. UX/UI 설계 기준 정리

## 목표

- Phase 5 기능 구현 전에 MVP 전체 사용자 흐름의 UX/UI 기준을 결정 완료 상태로 고정한다.
- 홈 검색 시작, 도서 선택, 지역 선택, 결과 확인까지의 화면 구조와 정보 위계를 문서화한다.
- 라이트/다크 테마, 색상, 타이포, 간격, 상태 피드백, 반응형 기준을 공통 계약으로 정리한다.
- Phase 5 구현자가 추가 디자인 결정을 하지 않고도 바로 화면 구현에 들어갈 수 있는 design handoff spec을 만든다.

## 기술 결정

- 디자인 컨텍스트 source of truth
  - 루트 `.impeccable.md`
  - 루트 `AGENTS.md`의 `Design Context`
- 설계 톤: 절제된 게임감
- 설계 범위: MVP 전체 화면
- 산출물 성격: 문서 설계 전용, 코드 구현은 포함하지 않음
- Phase 5 handoff 기준
  - screen inventory
  - state matrix
  - layout contract
  - theme/token contract
  - interaction/motion contract
  - copy and CTA contract

## 구현 범위

- 홈 검색 시작 화면의 첫 행동 구조와 핵심 CTA를 정의한다.
- 도서 검색 결과와 도서 선택 화면의 정보 구조를 정의한다.
- 지역 선택 다이얼로그의 단계, 활성/비활성 규칙, 상호작용 흐름을 정의한다.
- 도서관 결과 화면의 지도/목록 조합과 화면별 레이아웃 계약을 정의한다.
- 로딩, 빈 상태, 오류 상태를 전체 플로우 기준으로 정리한다.
- 라이트/다크 테마, 토큰, 타이포, 간격, 밀도, 모션, 접근성 기준을 정리한다.
- HomePage audit에서 드러난 설계 문제를 문서 단계에서 선제적으로 해소한다.
- Phase 5 구현을 위한 `task.md` 작성 기준을 함께 정리한다.

## 비범위

- `apps/web`의 실제 페이지/컴포넌트 구현
- 전역 스타일 토큰과 공통 UI 컴포넌트의 실제 수정
- 카카오맵 SDK 연결, API 호출, 기능 상태 관리 구현
- Phase 5 feature/task를 대신 수행하는 코드 작업
- 디자인 툴 원본 제작이나 외부 프로토타이핑 툴 문서화

## 현재 기반 상태

- 루트 `.impeccable.md`와 `AGENTS.md`에 사용자, 브랜드, 미감, 접근성 기준이 정리돼 있다.
- 현재 `HomePage`는 설명 중심 placeholder 성격이 강하고, 실제 첫 행동 UI가 없다.
- 현재 전역 스타일은 밝은 배경, 블루 액센트, 둥근 패널, 카드 중심 구조를 사용하고 있다.
- `HomePage` audit 기준으로 다음 문제가 이미 확인됐다.
  - 첫 행동 UI 부재
  - 중첩 카드와 반복 카드 패턴
  - 보조 텍스트 대비 부족
  - 다크 모드 계약 부재
- `entities/book`, `entities/library`, `entities/region`는 Phase 5 조합에 필요한 읽기 모델을 이미 제공한다.

## Design Context 운영 기준

- Phase 4-2 설계의 1차 source of truth는 루트 `.impeccable.md`다.
- `AGENTS.md`의 `Design Context`는 작업 규칙용 mirror로 유지한다.
- 두 문서의 표현이나 세부 기준이 충돌하면 `.impeccable.md`를 우선 적용한다.
- `.impeccable.md`가 갱신되면 `AGENTS.md`와 Phase 4-2 문서도 같은 턴 또는 후속 문서 작업에서 동기화한다.
- Phase 4-2 문서는 위 두 문서 바깥의 새 브랜드 가정을 임의로 추가하지 않는다.

## UX/UI 상세 설계

### 1. 경험 원칙

- 첫 행동은 즉시 보여야 한다.
- 설명보다 조작을 먼저 배치한다.
- 각 화면은 하나의 핵심 질문과 하나의 핵심 CTA를 가진다.
- 게임 같은 느낌은 진행감, 선택 피드백, 결과 도달감에만 제한적으로 사용한다.
- 실제 업무성 정보 전달은 항상 명확하고 과장되지 않게 유지한다.

### 1-1. `절제된 게임감` 경계

- 허용하는 표현
  - 단계 진행이 분명하게 느껴지는 화면 전환
  - 선택 완료 직후의 짧은 피드백
  - 결과 도달 시의 가벼운 보상감과 시선 유도
- 허용하지 않는 표현
  - 장식 목적의 진행 바, 레벨 UI, 업적 UI
  - 정보 전달보다 앞서는 과한 배지/스티커식 장식
  - bounce, elastic, 과도한 scale 같은 과장 모션
  - 실용성보다 “게임처럼 보이는 것”을 우선하는 레이아웃
- 즉, Phase 4-2에서 말하는 `절제된 게임감`은 실용적인 UX 위에 얹는 상태 피드백의 성격이지, 게임 인터페이스 자체를 차용하는 방향이 아니다.

### 2. Screen Inventory

- `Home / Search Start`
  - 목적: 제목 또는 저자 기준으로 검색을 즉시 시작하게 한다
  - 핵심 CTA: 검색 실행
  - 핵심 조작: `책 제목` / `저자명` 세그먼트 탭 + 단일 입력 필드
  - 입력 구조 순서: 세그먼트 탭 → 단일 입력 필드 → 검색 CTA
  - 기본값: `책 제목`
  - 전환 규칙: 세그먼트 전환 시 입력값 유지
  - 보조 요소: 짧은 안내 1문단 + 예시 검색어 2~4개
- `Book Search Result / Book Select`
  - 목적: 검색 결과 중 실제 찾는 도서를 빠르게 판별하게 한다
  - 핵심 CTA: 각 도서 카드의 즉시 선택 CTA
  - 핵심 요소: 검색어 기준 도서 목록, 도서 메타 정보
  - 상태: 검색 결과 없음 상태
- `Region Selection Dialog`
  - 목적: 선택한 도서를 기준으로 조회할 지역을 확정하게 한다
  - 핵심 CTA: 지역 선택 완료
  - 핵심 요소: 시/도 선택, 세부 지역 선택
  - 상태: 지역 결과 없음 상태
- `Library Result`
  - 목적: 지도와 상세 정보를 함께 보며 갈 도서관을 비교하고 선택하게 한다
  - 핵심 CTA: 특정 도서관을 선택하고 상세 정보를 확인한다
  - 데스크톱 기본 구조: 지도 2/3 + 선택된 도서관 정보 패널 1/3
  - 핵심 요소: 도서 요약 정보, 지도, 마커, 선택된 도서관 정보 패널
  - 패널 최소 정보: `name`, `address`, `phone`, `operatingTime`, `closedDays`
  - 상태: 도서관 결과 없음 상태
- `Shared States`
  - 목적: 흐름을 끊지 않고 현재 상태와 다음 행동을 안내한다
  - 핵심 CTA: 복구 또는 다음 행동
  - 상태: 전체 화면 로딩, 부분 로딩, API 오류, 잘못된 검색/선택 조합

### 3. Flow Contract

- 전체 흐름은 route 분리형이 아니라 단일 페이지 단계형을 기본으로 하고, 지역 선택만 다이얼로그로 처리한다.
- `Home / Search Start`
  - 진입 조건: 홈 첫 진입 또는 검색 초기화 상태
  - 다음 단계 조건: 유효한 제목 또는 저자명 입력 후 검색 실행
  - 핵심 행동: 책 제목 또는 저자명을 입력하고 검색을 시작한다
- `Book Search Result / Book Select`
  - 진입 조건: 검색 결과가 1건 이상 반환됨
  - 다음 단계 조건: 사용자가 명시적으로 도서를 선택하고 확정함
  - 핵심 행동: 결과 목록에서 찾을 도서를 확인하고 선택한다
  - 검색 결과가 1건이어도 자동 진행하지 않고 이 단계를 유지한다
- `Region Selection Dialog`
  - 진입 조건: 도서 선택이 확정됨
  - 다음 단계 조건: `region`이 선택됨
  - 핵심 행동: 도서관을 찾을 지역을 선택한다
- `Library Result`
  - 진입 조건: 도서와 지역 조합이 확정되고 조회가 완료됨
  - 다음 단계 조건: 재검색, 도서 재선택, 지역 재선택 중 하나
  - 핵심 행동: 지도와 상세 정보를 함께 보며 갈 도서관을 비교하고 판단한다
- 이전 단계 이동 시 검색어, 선택한 도서, 선택한 지역은 가능한 범위에서 유지한다.
- 도서를 다시 선택하더라도 검색어는 유지한다.
- 결과 화면에서 지역 선택을 다시 열어도 직전 지역 선택 상태는 유지한다.
- 결과 화면은 지도와 상세 정보를 함께 제공하되, 모바일에서는 순차 정보 구조를 우선한다.

### 4. Screen Contract

#### Home / Search Start

- 첫 화면 최우선 요소는 설명 배너가 아니라 제목/저자 검색 입력이다.
- 검색 시작 구조는 체크박스나 셀렉트가 아니라 `책 제목` / `저자명` 세그먼트 탭 + 단일 입력 필드로 고정한다.
- 검색 입력 블록의 순서는 `세그먼트 탭 → 단일 입력 필드 → 검색 CTA`로 고정한다.
- 기본 선택값은 `책 제목`이며, 세그먼트 전환 시 입력값은 유지한다.
- 헤드라인은 사용자의 과업을 바로 설명해야 하며, 부연 문구는 한 문단 이내로 제한한다.
- 보조 설명 영역은 짧은 안내 1문단과 예시 검색어 2~4개까지만 허용한다.
- 예시 검색어는 입력 유도를 돕는 보조 요소이며, 별도 카드 섹션이나 독립 콘텐츠 블록으로 확장하지 않는다.
- 검색 시작점은 사용자가 책 제목을 정확히 모를 때도 저자명으로 접근할 수 있어야 한다.
- 검색 입력보다 시각적 비중이 큰 소개 섹션, 사용 방법 섹션, 단계 설명 섹션은 홈 메인 구조에 두지 않는다.
- 반복 카드 3개를 동일 비중으로 나열하는 패턴은 사용하지 않는다.
- 카드 안에 빈 상태 카드가 다시 들어가는 중첩 카드 패턴은 사용하지 않는다.
- “어떻게 동작하는지” 설명은 보조 영역으로 축소한다.
- 빈 상태 카드로 랜딩 화면을 대체하지 않는다.

#### Book Search Result / Book Select

- 사용자는 검색 결과에서 “어떤 책을 선택해야 하는지” 빠르게 판단할 수 있어야 한다.
- 각 아이템의 판단 정보 우선순위는 `title → author → publisher/publicationYear → isbn13/loanCount`로 고정한다.
- 각 아이템은 작은 표지 썸네일, 제목, 저자, 출판 관련 보조 정보, 선택 CTA를 가진다.
- 표지는 보조 썸네일로만 사용하며, 카드의 주인공은 제목과 저자 같은 텍스트 정보다.
- 출판사와 출판연도는 동명이본 구분을 돕는 보조 정보로 사용한다.
- ISBN13과 대출 횟수는 필요 시 작은 보조 텍스트 수준으로만 다룬다.
- 선택 CTA는 각 카드 안에서 바로 보이는 즉시 선택 구조로 고정한다.
- 검색 결과가 1건이어도 같은 카드 구조를 유지하고, 사용자가 직접 CTA를 눌러 확정한다.
- 동일한 카드 반복이 필요하더라도 선택 우선순위가 시각적으로 드러나야 한다.
- 검색 결과 없음 상태에서는 현재 검색어와 검색 기준을 유지한 채 같은 화면에서 재검색하도록 유도한다.
- 검색 결과 없음 상태의 핵심 CTA는 홈 복귀가 아니라 `검색어 수정 후 다시 검색`이다.
- 결과 없음 상태에서는 짧은 안내와 예시 검색어 2~3개를 보조 요소로 둘 수 있지만, 주된 복구 경로는 현재 화면에서의 재검색이어야 한다.
- 모바일에서는 목록 집중, 데스크톱에서는 선택과 보조 설명을 병행할 수 있다.

#### Region Selection Dialog

- 지역 선택은 `상위 지역 필수 + 세부 지역 선택적 refinement` 구조로 고정한다.
- 상위 지역 선택 전에는 세부 지역 목록과 선택 완료 CTA가 비활성 상태여야 한다.
- 상위 지역을 선택하면 세부 지역 목록이 활성화되고, 첫 옵션은 항상 `전체`여야 한다.
- 상위 지역 선택 직후 세부 지역의 기본 선택값은 `전체`이며, 이 상태는 `서울 전체`, `충북 전체` 같은 광역 검색을 의미한다.
- 상위 지역을 바꾸면 직전 세부 지역 선택은 버리고, 새 상위 지역의 `전체`로 즉시 초기화한다.
- 세부 지역은 선택 사항이므로, 선택 완료 CTA는 `region`이 선택되는 즉시 활성화한다.
- 현재 선택 상태는 선택된 행 강조와 하단 요약 바를 함께 사용해 분명히 드러나야 한다.
- 요약 바는 `서울 전체`, `충북 전체`, `서울 > 마포구`처럼 현재 검색 범위를 항상 명시해야 한다.
- 세종처럼 세부 지역이 1개뿐인 지역도 기본값은 `전체`를 유지하고, 필요할 때만 유일한 세부 지역으로 좁힐 수 있게 한다.
- `청주시 전체` 같은 중간 집계 옵션은 MVP 범위에 포함하지 않는다.
- 세부 지역 데이터가 없거나 노출 가능한 세부 지역이 없으면 `전체`만 유지하고, `세부 지역 없이 이 지역 전체를 검색합니다` 같은 보조 문구를 노출한다.
- 이 경우에도 상위 지역 전체 검색은 가능해야 하므로 선택 완료 CTA는 계속 활성 상태를 유지한다.
- 단계감은 주되, 과도한 게임 UI나 장식적 진행 바는 도입하지 않는다.

#### Library Result

- 화면의 주인공은 “지도”가 아니라 “어느 도서관을 갈지 판단하는 정보”다.
- 데스크톱 기본 레이아웃은 `왼쪽 도서관 리스트 1/3 + 오른쪽 결과 패널 2/3` 구조로 고정한다.
- 오른쪽 결과 패널 내부는 `상단 지도 2/3 + 하단 선택된 도서관 정보 1/3` 구조로 고정한다.
- 결과가 1건 이상이면 첫 번째 도서관을 기본 선택 상태로 두고 화면에 진입한다.
- 왼쪽 리스트는 후보 탐색의 주 수단이며, 각 아이템은 최소 `name`, `address`를 우선 표시해야 한다.
- 오른쪽 지도는 검색된 모든 도서관의 마커를 표시하고, 현재 선택된 도서관만 시각적으로 강조해야 한다.
- 좌표가 없는 도서관은 리스트와 정보 패널에는 남기되, 지도 마커에서는 제외한다.
- 하단 정보 패널은 선택된 도서관의 상세 정보 패널로 동작해야 하며, 정보 우선순위는 `name → operatingTime/closedDays → address/phone`으로 고정한다.
- `homepage`는 보조 링크 수준으로만 취급하고, `fax`는 기본 우선순위에서 제외한다.
- 왼쪽 리스트에서 도서관을 선택하면 지도 강조 마커와 하단 정보 패널이 함께 갱신돼야 한다.
- 지도 마커를 선택해도 리스트 선택 상태와 하단 정보 패널이 같은 선택 상태로 함께 갱신돼야 한다.
- 모바일 기본 구조는 `도서 요약 → 선택된 도서관 정보 패널 → 지도` 순으로 고정한다.
- 모바일에서도 지도와 정보 패널은 같은 데이터를 서로 다른 방식으로 보여줘야 한다.
- 도서관 결과 없음 상태에서는 리스트/지도/정보 패널 대신 empty state를 보여주고, 주된 복구 행동은 `지역 다시 선택`, 보조 복구 행동은 `다른 책 다시 선택`으로 고정한다.

### 5. State Matrix Contract

- `loading`
  - 모든 주요 로딩 상태는 spinner 단독이 아니라 `레이아웃 skeleton + 짧은 단계 안내` 조합으로 고정한다.
  - 다음 행동이 기다림 상태라는 점과, 지금 무엇을 불러오고 있는지 한 줄 안에서 분명해야 한다.
  - 도서 검색 로딩은 결과 목록 골격을 먼저 보여주고, `도서를 찾고 있습니다` 같은 짧은 단계 안내를 함께 노출한다.
  - 도서관 결과 로딩은 결과 화면 골격을 먼저 보여주고, `선택한 지역의 도서관을 찾고 있습니다` 같은 짧은 단계 안내를 함께 노출한다.
  - 장문의 로딩 설명이나 전체 화면을 비우는 단일 spinner 패턴은 기본값으로 사용하지 않는다.
- `empty`
  - “아무것도 없음”이 아니라 현재 조건에서 결과가 없다는 사실과 다음 행동을 함께 안내해야 한다.
  - 도서 검색 결과 없음 상태는 현재 검색어와 검색 기준을 유지한 채, `검색어 수정 후 다시 검색`을 주 복구 행동으로 제공한다.
  - 지역 결과 없음 상태는 세부 지역 데이터가 없더라도 상위 지역 전체 검색 fallback을 유지하고, 필요 시 다른 상위 지역 선택으로 복구시킨다.
  - 도서관 결과 없음 상태는 현재 선택한 책과 지역 범위를 유지한 채, `지역 다시 선택`을 주 복구 행동으로, `다른 책 다시 선택`을 보조 복구 행동으로 제공한다.
  - 세 empty state는 모두 서로 다른 메시지와 서로 다른 복구 CTA를 가져야 한다.
- `error`
  - 검색, 선택, 결과 조회 단계에서 발생한 일반적인 요청 오류는 흐름 내부의 복구 UI로 우선 처리한다.
  - 오류 UI는 `무슨 문제가 생겼는지`, `지금 할 수 있는 복구 행동`, `이전 단계 대안`을 함께 보여줘야 한다.
  - 공통 주 복구 행동은 `다시 시도`로 고정한다.
  - 도서 검색 단계의 보조 복구 행동은 `검색어 수정`, 결과 화면 단계의 보조 복구 행동은 `지역 다시 선택` 또는 `다른 책 다시 선택`으로 둔다.
  - 라우트 자체가 깨지거나 단계 맥락을 유지할 수 없는 예외적 오류만 error boundary / route error page로 위임한다.
- `disabled`
  - 비활성 상태의 제약 이유는 해당 컨트롤 바로 아래의 짧은 설명으로 고정한다.
  - 지역 선택 전 세부 지역 목록 아래에는 `먼저 상위 지역을 선택해주세요` 같은 이유 설명을 붙인다.
  - 지역 미선택 상태의 완료 CTA 아래에는 `상위 지역을 선택하면 검색 범위를 확정할 수 있습니다` 같은 이유 설명을 붙인다.
  - 툴팁이나 hover 전용 설명은 기본 패턴으로 사용하지 않는다.

### 6. Theme / Token Contract

- 라이트와 다크 모두 설계 범위에 포함한다.
- 다음 토큰 그룹을 Phase 4-2에서 문서로 고정한다.
  - background
  - surface
  - surface-muted
  - text
  - text-muted
  - line
  - accent
  - accent-soft
  - focus
- 일반 본문 텍스트와 보조 텍스트는 WCAG AA 대비를 만족해야 한다.
- 보조 텍스트 토큰은 현재보다 더 높은 대비를 목표로 한다.
- 테마 전환 시 카드, 배경, 라인, 오버레이, 다이얼로그, 지도 보조 패널까지 함께 고려한다.

### 7. Typography / Spacing Contract

- 화면마다 다른 텍스트 크기 체계를 임의로 만들지 않는다.
- display, section, item title, body, caption 수준의 계층을 문서로 고정한다.
- 여백은 컴포넌트 단위가 아니라 화면 리듬 단위로 정의한다.
- 랜딩, 다이얼로그, 결과 화면 모두 동일한 밀도 체계를 공유해야 한다.

### 8. Interaction / Motion Contract

- 기본 경험은 빠르고 차분해야 한다.
- 모션은 상태 전환과 선택 피드백에만 사용한다.
- 과장된 bounce/elastic 계열 움직임은 사용하지 않는다.
- 절제된 게임감은 아래에 한정한다.
  - 단계 전환의 명확한 피드백
  - 선택 완료 시 짧은 보상감
  - 결과 도달 시 시선 유도
- reduced motion을 해치지 않는 수준으로 문서화한다.
- 모션은 장식보다 정보 전달을 우선하며, `절제된 게임감` 경계를 넘지 않는다.

### 9. Responsive Contract

- 데스크톱과 모바일은 같은 화면을 단순 축소하지 않는다.
- 모바일에서는 첫 행동과 현재 상태가 항상 fold 상단에서 파악돼야 한다.
- 결과 화면은 좁은 화면에서 지도와 목록이 충돌하지 않는 구조로 고정한다.
- `100vh` 중심 레이아웃은 모바일 브라우저 주소창 변화에 안전한 방식으로 재검토한다.

### 10. Accessibility Contract

- 모든 주요 텍스트는 WCAG AA 대비 이상을 목표로 한다.
- heading 구조는 시각 표현과 문서 구조가 일치해야 한다.
- 입력, 다이얼로그, 선택 상태는 키보드와 스크린리더 기준으로 설계한다.
- 포커스 링, 비활성 상태 설명, 에러 메시지 위치를 문서로 고정한다.

- MVP 전체 화면과 상태가 모두 문서로 정의돼 있다.
- HomePage audit의 P0/P1 문제를 설계 단계에서 해소하는 기준이 반영돼 있다.
- 라이트/다크, 타이포, 간격, 상태, 반응형, 접근성 규칙이 최소 구현 단위까지 명시돼 있다.
- Phase 5 구현자가 추가 디자인 결정을 하지 않아도 되는 수준으로 문서가 결정 완료 상태다.

## 테스트 기준

- 문서만 읽고 각 화면의 핵심 CTA를 설명할 수 있어야 한다.
- 문서만 읽고 검색, 도서 선택, 지역 선택, 결과 확인 플로우를 이어서 구현할 수 있어야 한다.
- 문서만 읽고 로딩/빈 상태/오류 상태를 서로 다르게 설계할 수 있어야 한다.
- `.impeccable.md`와 상충되는 브랜드/미감 기준이 없어야 한다.
