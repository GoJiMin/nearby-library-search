# Phase 3. 모노레포 및 BFF 구성

## 목표

- 현재 단일 웹 앱 구조를 `pnpm` workspace 기반 모노레포로 전환한다.
- 웹 앱과 BFF의 책임을 분리해 외부 Open API 인증키가 클라이언트에 노출되지 않도록 한다.
- `apps/web`, `apps/bff`, `packages/contracts`를 기준으로 이후 `entities`, `features` 구현이 자연스럽게 이어질 수 있는 기반을 마련한다.
- BFF는 단순 프록시를 넘어서 이후 인증, 북마크, 추천, 데이터베이스 연동까지 확장 가능한 구조를 목표로 한다.

## 기술 결정

- 모노레포 패키지 매니저: `pnpm workspace`
- 웹 앱 위치: `apps/web`
- BFF 위치: `apps/bff`
- BFF 프레임워크: `Fastify`
- 공통 계약 패키지: `packages/contracts`
- 웹 앱과 BFF 언어: `TypeScript`
- 외부 Open API 호출 위치: `apps/bff`만 허용
- 외부 Open API 인증키 저장 위치: BFF 서버 환경변수만 허용

## 구현 범위

- 루트 프로젝트를 `pnpm` workspace 구조로 전환한다.
- 현재 웹 앱 코드를 `apps/web` 기준으로 이동 가능한 구조로 재정리한다.
- `apps/bff`에 Fastify 기반 서버 엔트리와 기본 앱 구조를 만든다.
- `packages/contracts`에 웹 앱과 BFF가 함께 사용하는 요청/응답 타입 또는 스키마 구조를 만든다.
- 현재 서비스에서 사용하는 3개 Open API를 BFF 관점에서 수용할 수 있는 호출 규칙과 파싱 기준을 정리한다.
- 웹 앱이 외부 Open API가 아니라 BFF 엔드포인트만 호출하도록 요청 경계를 정리한다.
- 외부 Open API 호출 시 BFF가 내부 계약으로 응답을 정규화하는 기준을 정의한다.
- 로컬 개발, 빌드, 배포에서 웹 앱과 BFF를 분리해 다루는 기본 흐름을 정리한다.
- 이후 Phase 4 `entities`가 BFF 계약을 기준으로 개발될 수 있도록 진입 구조를 마련한다.

## 비범위

- 실제 도서 Open API 세부 파라미터와 엔드포인트 구현
- 실제 인증 기능 구현
- 북마크, 추천, 사용자 프로필 기능 구현
- 데이터베이스 도입과 마이그레이션 구성
- 배포 플랫폼 최종 확정
- 모니터링, 로그 집계, 운영 자동화 고도화

## 디렉터리 기준

- `apps/web`
  - 기존 React 앱을 옮길 웹 클라이언트 패키지
  - 기존 `src` 기준 FSD 구조를 유지하되 패키지 경계만 분리한다
- `apps/bff`
  - Fastify 서버 패키지
  - 외부 Open API 호출, 서버 환경변수 관리, 응답 정규화, 이후 인증/DB 진입점을 담당한다
- `packages/contracts`
  - 웹 앱과 BFF가 함께 사용하는 요청/응답 타입 또는 스키마를 관리한다
  - 외부 Open API raw response가 아니라 내부 계약만 이 패키지에 둔다

## 아키텍처 상세 설계

### 1. 모노레포 구조

- 루트는 workspace 설정과 공통 스크립트만 관리한다.
- 웹 앱과 BFF는 독립 패키지로 분리하되, 같은 저장소에서 함께 개발한다.
- 공통 계약은 `packages/contracts`로 분리해 웹 앱과 BFF가 동일한 타입 기준을 사용한다.
- 웹 앱 전용 설정과 BFF 전용 설정은 각 패키지 내부에서 관리한다.

### 2. 웹 앱 경계

- 웹 앱은 외부 Open API를 직접 호출하지 않는다.
- 웹 앱은 `shared/request`를 통해 BFF 엔드포인트만 호출한다.
- 웹 앱의 `VITE_API_BASE_URL`은 외부 provider가 아니라 Fastify BFF base URL만 가리킨다.
- `shared/request`는 `/api` 네임스페이스의 BFF 애플리케이션 엔드포인트만 호출한다.
- 웹 앱은 외부 인증키를 포함한 서버 전용 설정을 알지 못한다.
- 이후 `entities` 레이어는 외부 Open API가 아니라 BFF 응답 계약만 기준으로 구현한다.

### 3. BFF 경계

- BFF는 외부 Open API 인증키를 서버 환경변수에서 읽는다.
- BFF는 외부 Open API 호출을 담당한다.
- BFF는 외부 응답을 그대로 전달하지 않고, 웹 앱이 사용하는 내부 계약으로 정규화해 반환한다.
- 이후 인증, DB, 사용자 기능이 들어오더라도 BFF 내부에서 흡수할 수 있게 구조를 잡는다.
- MVP 단계에서는 과도한 레이어 분리보다 명확한 요청 경계와 확장 가능한 엔트리 구조를 우선한다.

### 4. 외부 Open API 호출 규칙

- 현재 BFF가 우선 수용해야 하는 외부 Open API는 다음 3개다.
  - 도서 검색
  - 도서 상세 조회
  - 도서 소장 도서관 조회
- 모든 외부 Open API 요청은 `GET` 방식으로 호출한다.
- 모든 외부 Open API 요청에는 `authKey`, `format=json` 쿼리를 반드시 포함한다.
- 검색 조건이 없는 파라미터는 쿼리에 포함하지 않는다.
- 필수 파라미터가 없는 경우 외부 Open API를 호출하지 않는다.
- BFF는 외부 Open API 호출 전에 필수 파라미터 검증을 선행한다.
- 외부 Open API 호출 실패 시 웹 앱에는 재시도 가능한 에러 메시지를 노출할 수 있는 형태로 정규화한다.

### 5. 응답 파싱 규칙

- 실제 데이터는 모두 `response` 아래에 있다고 가정하고 파싱한다.
- 배열 내부 한 단계 중첩 구조를 주의해 파싱한다.
  - 예: `docs[].doc`
  - 예: `libs[].lib`
- 응답 스키마가 일부 누락될 수 있으므로 optional chaining 기반으로 안전하게 파싱한다.
- 빈 문자열(`""`)은 nullable 또는 fallback 기준을 통해 후처리한다.
- 검색 결과가 0건이면 웹 앱이 빈 상태 UI를 표시할 수 있는 형태로 반환한다.
- 일부 숫자 필드는 문자열로 내려올 수 있으므로 원본 응답 단계에서는 문자열 기반으로 우선 처리한다.

### 6. 타입 설계 및 공통 계약 패키지

- `packages/contracts`에는 웹 앱과 BFF가 함께 사용하는 타입 또는 스키마만 둔다.
- 외부 Open API의 raw 타입은 이 패키지에 직접 노출하지 않는다.
- 내부 계약은 이후 기능 확장 시에도 비교적 안정적으로 유지될 수 있는 단위로 정의한다.
- 요청/응답 계약, 공통 에러 응답 구조, 공통 식별자 타입 등을 이 패키지 후보로 본다.
- 공통 에러 응답은 `title`, `detail`, `status` 형식으로 표준화한다.
- `title`은 클라이언트가 분기 가능한 에러 코드로 사용한다.
- `detail`은 사용자 노출 또는 로깅에 사용할 상세 메시지로 사용한다.
- `status`는 HTTP 상태 코드를 본문에도 명시해 클라이언트가 일관되게 해석할 수 있게 한다.
- API 원본 응답 타입과 웹 앱 UI용 모델 타입은 분리한다.
- Open API raw 응답 타입은 `apps/bff` 내부에서만 관리하고 `packages/contracts`에 노출하지 않는다.
- `packages/contracts`에는 웹 앱이 직접 사용하는 정규화된 응답 계약만 둔다.
- `BookSearchResponse`, `BookDetailResponse`, `LibrarySearchResponse`는 웹 앱이 소비하는 BFF 계약 타입이다.
- `docs[].doc`, `detail[].book`, `libs[].lib` 같은 외부 응답 중첩 구조는 BFF 파싱 레이어 내부에만 머문다.
- 문자열 숫자 파싱, 빈 문자열 후처리, nullable 정리, 좌표 숫자 변환은 BFF 정규화 단계에서 수행한다.
- UI에서는 실제 필요한 필드만 정제한 `Model` 타입을 사용한다.
- ISBN, ISBN13, 지역코드, 세부지역코드는 모두 `string`으로 다룬다.
- ISBN은 숫자처럼 보여도 앞자리 0 보존과 일관성 유지를 위해 `string` 기준으로 처리한다.
- 일부 숫자형 값은 외부 응답에서는 `string`으로 받고, 필요 시 내부 모델 변환 단계에서 숫자로 변환한다.
- `region`, `dtl_region`처럼 여러 엔드포인트에서 공통으로 쓰는 스키마는 `packages/contracts`에 둘 수 있게 설계한다.

### 7. 환경변수 규칙

- 웹 앱 공개 환경변수는 `VITE_` 접두사를 유지한다.
- `VITE_API_BASE_URL`은 웹 앱이 호출할 Fastify BFF 주소로만 사용한다.
- BFF 서버 전용 시크릿은 `VITE_` 접두사를 사용하지 않는다.
- 외부 Open API 인증키는 BFF 런타임 환경변수에서만 읽는다.
- 웹 앱 코드, 공통 계약 패키지, 프론트 번들에는 외부 Open API 인증키가 포함되지 않아야 한다.

### 8. 개발 흐름

- 로컬 개발 시 웹 앱과 BFF를 함께 실행한다.
- 웹 앱은 개발 환경에서 BFF의 로컬 주소를 바라보도록 구성한다.
- 루트에서 웹 앱과 BFF를 동시에 실행할 수 있는 공통 스크립트 구성이 필요하다.
- 개발 중에는 웹 앱과 BFF를 독립적으로 테스트하고 빌드할 수 있어야 한다.

### 9. 빌드 및 배포 관점

- 웹 앱과 BFF는 별도 빌드 단위를 가진다.
- 배포 시에도 웹 앱과 BFF를 분리해 다루는 기준을 전제로 한다.
- 현재 Phase에서는 특정 배포 플랫폼을 확정하지 않지만, 전통적인 Node 서버 배포가 가능한 구조를 우선한다.
- 이후 배포 방식이 바뀌더라도 웹 앱과 BFF 경계가 유지되도록 구조를 설계한다.

## 완료 기준

- 저장소가 `apps/web`, `apps/bff`, `packages/contracts` 기준 구조로 정리된다.
- 웹 앱은 외부 Open API를 직접 호출하지 않고 BFF만 호출하는 방향으로 정리된다.
- 외부 Open API 인증키는 BFF 서버 환경변수에서만 관리된다.
- BFF는 외부 API 응답을 내부 계약으로 정규화할 수 있는 구조를 가진다.
- 외부 Open API의 공통 호출 규칙과 안전한 파싱 규칙이 문서화된다.
- 웹 앱과 BFF가 공통 계약 패키지를 통해 요청/응답 타입을 공유할 수 있다.
- 로컬 개발과 빌드 시 웹 앱과 BFF를 각각 또는 함께 실행할 수 있는 기본 흐름이 정리된다.

## 테스트 기준

- 웹 앱과 BFF가 각각 타입 체크와 빌드를 통과한다.
- 공통 계약 패키지가 웹 앱과 BFF에서 동시에 정상 참조된다.
- 웹 앱 코드에 외부 Open API 직접 호출이 남아 있지 않다.
- BFF의 최소 헬스체크 또는 샘플 엔드포인트가 정상 응답한다.
- 외부 Open API 인증키가 웹 앱 번들에 포함되지 않는 구조가 확인된다.
- 필수 파라미터가 없을 때 외부 Open API를 호출하지 않는 구조가 확인된다.
- 0건 응답과 실패 응답이 웹 앱에서 빈 상태 또는 재시도 가능 에러로 이어질 수 있는 구조가 확인된다.

## 후속 연결 포인트

- Phase 4에서는 `packages/contracts`와 BFF 엔드포인트를 기준으로 `entities` 요청 함수와 데이터 모델을 설계한다.
- Phase 5에서는 BFF 기반 ISBN 조회, 지역 선택, 도서관 조회 기능을 실제 사용자 흐름으로 연결한다.

## 기본 가정

- 현재 저장소는 아직 웹 앱 단일 구조이며, Phase 3에서 모노레포로 전환한다.
- BFF는 초기에는 외부 Open API 프록시와 응답 정규화 역할을 우선한다.
- 이후 인증, 북마크, 추천, DB 기능이 추가될 가능성을 고려해 Fastify를 채택한다.
- 실제 Open API 세부 스펙은 `open_api_spec.md`를 기준으로 하고, 이후 구현 단계에서 엔드포인트별 상세 타입을 추가 보완한다.
