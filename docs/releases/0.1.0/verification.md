# 0.1.0 Verification

이 문서는 `0.1.0`을 왜 배포 가능한 상태로 판단했는지 설명하는 기록입니다. 핵심은 단순합니다. 사용자가 실제로 쓰게 될 흐름이 지금도 이어지고 있고, 그 흐름을 받치는 web과 bff의 경계가 다시 깨지지 않았으며, 운영 주소에서도 같은 흐름이 실제로 응답하는지를 다시 확인했습니다.

## 이번 버전을 배포 가능하다고 본 이유

이번 확인에서 가장 중요하게 본 것은 “사용자가 책을 찾고, 가까운 도서관을 보고, 대출 가능 여부까지 확인하는 흐름이 끊기지 않는가”였습니다. `0.1.0`은 이 흐름이 완성된 첫 MVP이기 때문에, 개별 화면보다 이 연결이 유지되는지가 더 중요합니다.

이번 Task 4에서 다시 확인한 결론은 아래와 같습니다.

- 검색 시작부터 `/books` 결과 진입, 재검색, 페이지 이동까지의 기본 흐름이 다시 깨지지 않았습니다.
- 책 상세 보기, 지역 선택, 도서관 결과 다이얼로그, 대출 가능 여부 조회까지의 후속 흐름도 다시 이어졌습니다.
- bff는 정상 요청뿐 아니라 외부 도서 Open API가 불안정하게 응답하는 경우까지 현재 기준의 오류 형태로 다시 정리했습니다.
- 운영 주소 `nearlib.com`, `api.nearlib.com`에서도 검색, 상세, 도서관 검색, 대출 가능 여부 조회가 실제로 응답했습니다.

즉 `0.1.0`은 “코드는 있는데 아직 흐름이 불안정한 상태”가 아니라, 적어도 지금 정의한 MVP 흐름은 다시 밟을 수 있는 상태라고 판단했습니다.

## 사용자가 겪는 핵심 흐름은 어디까지 다시 확인했는가

web 쪽에서는 사용자 입장에서 중요한 흐름을 다시 확인했습니다. 단순히 버튼이 보이는지 수준이 아니라, 주소와 화면 상태가 이어지는지를 중점적으로 봤습니다.

이번에 다시 확인한 범위는 아래와 같습니다.

- 홈에서 책 제목 또는 저자명으로 검색을 시작할 수 있는지
- `/books` 결과 화면이 주소 기준으로 다시 열리는지
- 결과 화면 안에서 재검색과 페이지 이동이 이어지는지
- `상세 보기`가 현재 구조대로 열리는지
- `소장 도서관 찾기` 이후 지역 선택이 이어지는지
- 도서관 결과 다이얼로그가 리스트와 지도 기준으로 다시 열리는지

이 재확인에는 web 테스트 5개 파일, 144개 테스트가 포함됐습니다. 수치보다 중요한 건, 테스트가 지금 서비스의 중심 흐름에 닿고 있다는 점입니다. `0.1.0`에서 가장 중요한 사용 경험은 “책 검색 후 후속 선택이 자연스럽게 이어지는가”인데, 이번 검증은 바로 그 부분을 다시 확인했습니다.

## 서버는 무엇을 다시 확인했는가

bff는 화면 뒤에서 외부 도서 Open API를 대신 호출하는 역할을 합니다. 그래서 여기서는 “정상 응답만 잘 받는가”보다 “외부 응답이 흔들릴 때도 서비스가 현재 약속한 방식으로 버티는가”가 중요합니다.

이번에 다시 확인한 내용은 아래와 같습니다.

- 책 검색 route가 정상 요청과 잘못된 요청을 다시 구분하는지
- 책 상세 route가 필요한 값을 다시 정리해 돌려주는지
- 도서관 검색과 대출 가능 여부 조회가 현재 계약 기준으로 응답하는지
- 외부 도서 Open API가 비정상 응답을 줄 때도, bff가 지금 기준의 오류 응답으로 다시 정리하는지

이 재확인에는 bff 테스트 4개 파일, 36개 테스트가 포함됐습니다. 이 숫자 자체보다 중요한 건, 지금 bff가 단순한 전달 계층이 아니라 **오류를 걸러내고 형태를 정리하는 경계**로 다시 검증됐다는 점입니다.

## 자동 테스트는 얼마나 넓게 닿고 있는가

`0.1.0` 시점에서 자동 테스트가 서비스 어디까지 닿고 있는지도 다시 기록했습니다. 이 수치는 합격선이 아니라 기준선입니다. 이후 `0.2.0` 이상에서 품질이 좋아졌는지, 혹은 오히려 줄었는지 비교하는 기준으로 보는 게 맞습니다.

### web

- 실행된 코드 범위: `91.65%`
- 갈림길이 있는 조건 범위: `85.82%`
- 함수 범위: `90.11%`
- 코드 줄 범위: `91.39%`

즉 web은 현재 MVP의 주요 화면 흐름, dialog 흐름, 공통 요청 처리와 오류 처리까지 꽤 넓게 테스트가 닿고 있습니다. 단순한 happy path만 보는 상태는 아닙니다.

### bff

- 실행된 코드 범위: `92.44%`
- 갈림길이 있는 조건 범위: `83.94%`
- 함수 범위: `99.20%`
- 코드 줄 범위: `92.34%`

즉 bff도 핵심 route, 응답 정리, fixture 경로, 오류 경로까지 상당히 넓게 다시 확인된 상태입니다. 특히 함수 기준이 매우 높다는 건, 현재 서버 로직의 주요 분기가 거의 전부 다시 호출됐다는 뜻에 가깝습니다.

## 운영 주소에서는 무엇을 다시 확인했는가

문서와 테스트만으로는 부족하므로, 실제 운영 주소에서도 핵심 경계를 다시 확인했습니다. 이번에는 브라우저를 직접 조작한 것이 아니라, 운영 web 주소와 운영 API 주소를 따라가며 지금도 응답이 살아 있는지 다시 확인했습니다.

다시 확인한 순서는 아래와 같습니다.

1. `https://nearlib.com`이 정상 응답하는지
2. `파친코` 검색 기준 `/books` 주소가 직접 열리는지
3. 책 검색 결과가 실제로 돌아오는지
4. 그 책의 상세 정보가 다시 열리는지
5. 같은 책을 기준으로 도서관 검색이 다시 되는지
6. 특정 도서관에서 대출 가능 여부가 다시 조회되는지

확인 결과는 아래와 같았습니다.

- 홈 주소 응답 성공
- `/books` 주소 직접 진입 성공
- 검색 응답 성공, 전체 결과 수 `13`
- 상세 응답 성공
- 도서관 검색 응답 성공, 전체 결과 수 `261`
- 대출 가능 여부 조회 응답 성공

즉 운영 기준에서도 현재 MVP의 핵심 흐름을 이루는 응답 경계는 살아 있습니다. 적어도 “배포는 되어 있지만 실제 운영 API 경로가 죽어 있는 상태”는 아니었습니다.

## 운영 기준에서 남아 있는 확인 포인트

이번 확인에서 바로 막히는 문제는 없었지만, 그대로 적어둘 필요가 있는 점은 있었습니다.

- 현재 홈 HTML의 `<title>`은 아직 `동네 도서관 찾기`로 응답했습니다.
  - 문서와 코드 기준 브랜드명은 `니어립`이므로, 최신 web 배포에 메타데이터가 실제로 반영됐는지는 다시 확인할 필요가 있습니다.
- 이 서비스는 여전히 외부 도서 Open API 품질에 영향을 받습니다.
  - 그래서 bff 테스트에서도 “이상한 응답을 어떻게 현재 기준 오류로 바꾸는가”를 중요하게 다시 확인했습니다.

이 두 가지는 `0.1.0`을 막는 이유라기보다, 운영에서 계속 봐야 할 관찰 포인트에 가깝습니다.

## 정리

`0.1.0`은 지금 기준에서 배포 가능한 상태입니다. 이유는 단순합니다.

- 사용자가 실제로 쓰는 핵심 흐름이 다시 이어졌고
- 그 흐름을 받치는 서버 경계도 다시 확인됐으며
- 자동 테스트가 화면과 서버 모두에 충분히 닿고 있고
- 운영 주소에서도 실제 응답이 살아 있었기 때문입니다

즉 이번 버전은 “형태만 갖춘 MVP”가 아니라, **검색에서 대출 가능 여부 확인까지의 핵심 가치가 다시 검증된 MVP**라고 보는 편이 맞습니다.
