# 0.1.0 Overview

이 문서는 `0.1.0` 시점의 니어립 서비스가 무엇을 제공하고, 어떤 흐름으로 동작하며, 왜 현재 구조를 택했는지를 빠르게 이해하기 위한 릴리즈 개요입니다. 새로 투입되는 개발자가 이 문서 하나만 읽고도 현재 MVP의 목적과 범위를 파악할 수 있도록 정리합니다.

## 0.1.0 소개

`0.1.0`은 현재 MVP 공개 가능 상태를 기준으로 정리한 첫 릴리즈 스냅샷입니다. 이 문서는 이미 분리 배포되는 `web`과 `bff` 구조를 전제로, 지금 사용자가 실제로 사용할 수 있는 기능과 그 배경을 설명합니다.

## 서비스가 해결하는 문제

이 서비스는 “원하는 책을 당장 사기 어려운 사람이 도서관에서 대신 빌려 읽고 싶을 때, 그 책을 가진 도서관이 근처에 있는지와 지금 바로 빌릴 수 있는지를 빠르게 확인하기 어렵다”는 문제에서 시작했습니다.

기획 배경도 여기에서 나왔습니다. 취업 준비 중에는 기술 서적이나 원하는 도서를 매번 사기엔 금전적으로 부담이 컸고, 그래서 주로 근처 도서관에서 책을 빌려 봤습니다. 하지만 원하는 도서를 소장한 도서관이 근처에 있는지, 심지어 지금 대출 가능한 상태인지까지 확인하려면 검색과 확인 과정을 여러 번 반복해야 했습니다. 니어립은 그 과정을 줄여, 책 검색부터 근처 도서관 확인과 대출 가능 여부 확인까지 한 흐름으로 이어지게 만드는 데 집중합니다.

## 주 사용자

- 책을 자주 사기보다 도서관에서 빌려 읽는 동네 주민
- 기술 서적이나 전공서를 자주 찾는 학생과 취업 준비생
- 원하는 책을 짧은 시간 안에 찾아야 하는 일반 독자

이 사용자들은 공통적으로 긴 탐색보다 빠른 확인을 원합니다. 그래서 서비스도 짧은 세션 안에 검색, 선택, 확인을 끝낼 수 있게 설계했습니다.

## 핵심 사용자 플로우

1. 사용자는 홈에서 책 제목이나 저자명으로 검색을 시작합니다.
2. `/books` 결과 화면에서 검색 결과를 확인하고 원하는 책을 고릅니다.
3. 결과 카드에서 `상세 보기`로 책 정보를 더 확인하거나, `소장 도서관 찾기`로 다음 단계로 넘어갑니다.
4. `소장 도서관 찾기`를 누르면 지역 선택 다이얼로그가 열리고, 사용자는 상위 지역과 필요하면 세부 지역까지 고릅니다.
5. 선택한 지역을 기준으로 도서관 결과 다이얼로그가 열리며, 사용자는 리스트와 지도를 함께 보면서 도서관을 비교합니다.
6. 특정 도서관을 선택한 뒤 `대출 가능 여부 조회`를 눌러 지금 바로 빌릴 수 있는지 확인합니다.

## 현재 MVP에서 가능한 핵심 행동

- 책 제목 또는 저자명으로 검색 시작
- URL 기반 검색 결과 재방문과 페이지 이동
- 검색 결과에서 책 상세 정보 확인
- 지역을 선택해 해당 책을 소장한 근처 도서관 탐색
- 리스트와 지도로 도서관 위치와 기본 정보를 함께 비교
- 특정 도서관의 대출 가능 여부 확인

## 현재 아키텍처 개요

- `apps/web`
  - 사용자가 직접 보는 화면과 상호작용을 담당합니다.
  - 검색 시작, 검색 결과, 지역 선택, 도서관 결과, 대출 가능 여부, 상세 보기 흐름이 여기서 이어집니다.
- `apps/bff`
  - 외부 도서 Open API 호출과 응답 정규화를 담당합니다.
  - web은 외부 API를 직접 호출하지 않고 항상 BFF의 `/api` 엔드포인트만 사용합니다.
- `packages/contracts`
  - web과 bff가 같은 요청/응답 기준을 공유하도록 돕는 내부 계약 패키지입니다.
  - 화면과 서버가 같은 데이터 구조를 기준으로 움직일 수 있게 합니다.

## 이 구조를 택한 이유

현재 구조는 단순히 패키지를 나누기 위해 만든 것이 아니라, 서비스가 안정적으로 동작하게 하기 위해 선택한 구조입니다.

우선 BFF를 둔 가장 큰 이유는 외부 Open API 인증키를 브라우저에 노출하지 않기 위해서입니다. 사용자가 보는 web은 외부 도서 Open API를 직접 호출하지 않고, BFF만 호출하도록 경계를 분리했습니다. 이렇게 하면 키 보호뿐 아니라 외부 응답 형식이 바뀌거나 품질이 들쭉날쭉할 때도 BFF 안에서 먼저 흡수할 수 있습니다.

또한 BFF는 외부 도서 Open API 응답을 그대로 넘기지 않고, web이 쓰기 쉬운 내부 계약으로 정규화합니다. 덕분에 화면 쪽은 외부 API의 복잡한 응답 구조를 직접 알 필요 없이 필요한 필드만 안정적으로 사용할 수 있습니다.

`packages/contracts`를 따로 둔 이유도 같습니다. web과 bff가 같은 요청/응답 기준을 공유하면, 화면과 서버가 서로 다른 가정을 갖고 움직이는 일을 줄일 수 있습니다. 결과적으로 현재 구조는 보안, 경계 분리, 유지보수성을 함께 확보하기 위한 선택입니다.
